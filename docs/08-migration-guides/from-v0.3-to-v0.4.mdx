# Migrate Go AI SDK v0.3.x to v0.4.0

The Go AI SDK v0.4.0 introduces breaking changes to streaming behavior, provider defaults, MCP security, and telemetry. This guide helps you migrate your existing v0.3.x code.

## Recommended migration process

Follow these five steps to upgrade:

1. **Ensure all tests pass on v0.3.x.** Run `go test ./...` before making any changes so you have a clean baseline.
2. **Update to v0.4.0:**
   ```bash
   go get github.com/digitallysavvy/go-ai@v0.4.0
   ```
3. **Follow the breaking changes below** to update your code.
4. **Build and test:**
   ```bash
   go build ./...
   go test ./...
   ```
5. **Commit** once everything passes.

## Breaking changes

### Streaming tool execution deferred

Tools passed to `StreamText` now execute **after** the stream is fully consumed, not mid-stream. In v0.3.x, tool-result chunks could arrive interleaved with text chunks. In v0.4.0, the stream delivers all provider chunks (text, reasoning, custom content) first, then executes tools and delivers tool-result chunks after the stream ends.

This ensures consumers receive all content before any tool side-effects fire. If your application depends on processing tool results mid-stream, restructure your logic to handle them after the stream completes.

**Before (v0.3.x):**

```go
result, err := ai.StreamText(ctx, ai.StreamTextOptions{
    Model:  model,
    Prompt: "What's the weather?",
    Tools:  []types.Tool{weatherTool},
    OnChunk: func(chunk provider.StreamChunk) {
        // v0.3.x: tool-result chunks could arrive between text chunks
        fmt.Print(chunk.Text)
    },
})
if err != nil {
    log.Fatal(err)
}
defer result.Close()

for range result.Chunks() {
}
```

**After (v0.4.0):**

```go
result, err := ai.StreamText(ctx, ai.StreamTextOptions{
    Model:  model,
    Prompt: "What's the weather?",
    Tools:  []types.Tool{weatherTool},
    OnChunk: func(chunk provider.StreamChunk) {
        // v0.4.0: all provider chunks arrive first,
        // then tool-result chunks after stream end.
        // Order: provider chunks -> stream end -> tool execution -> tool-result
        fmt.Print(chunk.Text)
    },
})
if err != nil {
    log.Fatal(err)
}
defer result.Close()

for range result.Chunks() {
}
```

### XAI default API changed to Responses API

`xai.Provider.LanguageModel()` now returns a model backed by the Responses API instead of the Chat Completions API. The Responses API supports additional features like image input and more granular reasoning effort levels. If you need the legacy Chat Completions behavior, use `ChatCompletionsLanguageModel()` instead.

**Before (v0.3.x):**

```go
import "github.com/digitallysavvy/go-ai/pkg/providers/xai"

provider := xai.New(xai.Config{APIKey: os.Getenv("XAI_API_KEY")})

// v0.3.x: LanguageModel() used the Chat Completions API
model, _ := provider.LanguageModel("grok-3")
```

**After (v0.4.0):**

```go
import "github.com/digitallysavvy/go-ai/pkg/providers/xai"

provider := xai.New(xai.Config{APIKey: os.Getenv("XAI_API_KEY")})

// v0.4.0: LanguageModel() now uses the Responses API (new default)
model, _ := provider.LanguageModel("grok-3")

// To explicitly use the Chat Completions API:
chatModel, _ := provider.ChatCompletionsLanguageModel("grok-3")
```

### XAI model IDs removed

`grok-2` and `grok-2-vision-1212` have been removed from the XAI provider. Use `grok-3` family models instead.

**Before (v0.3.x):**

```go
model, _ := provider.LanguageModel("grok-2")             // Removed
model, _ := provider.LanguageModel("grok-2-vision-1212") // Removed
```

**After (v0.4.0):**

```go
model, _ := provider.LanguageModel("grok-3")        // Use grok-3 family
model, _ := provider.LanguageModel("grok-3-mini")   // Or grok-3-mini
```

### MCP default redirect mode changed

MCP HTTP transport previously followed redirects by default. In v0.4.0, the default is `MCPRedirectError` ŌĆö redirects cause an error. This prevents SSRF attacks where a malicious MCP server redirects requests to internal services.

**Before (v0.3.x):**

```go
import "github.com/digitallysavvy/go-ai/pkg/mcp"

// v0.3.x: redirects followed automatically (potential SSRF risk)
transport := mcp.NewHTTPTransport(mcp.HTTPTransportConfig{
    URL: "https://mcp.example.com",
})
```

**After (v0.4.0):**

```go
import "github.com/digitallysavvy/go-ai/pkg/mcp"

// v0.4.0: redirects rejected by default (MCPRedirectError)
transport := mcp.NewHTTPTransport(mcp.HTTPTransportConfig{
    URL: "https://mcp.example.com",
})

// To restore old behavior (only if you trust the MCP server):
transport := mcp.NewHTTPTransport(mcp.HTTPTransportConfig{
    URL: "https://mcp.example.com",
    Config: mcp.TransportConfig{
        Redirect: mcp.MCPRedirectFollow,
    },
})
```

> **Security note:** Only set `Redirect: mcp.MCPRedirectFollow` when you trust the MCP server and have verified it does not redirect to internal services.

### Telemetry global registry

The `Tracer` field on `ExperimentalTelemetry` settings is no longer the recommended way to configure custom OpenTelemetry tracers. Instead, register a telemetry integration once at startup using `RegisterTelemetryIntegration()`. The `ExperimentalTelemetry` option still exists for controlling `IsEnabled`, `RecordInputs`, `RecordOutputs`, `FunctionID`, and `Metadata` per-call.

**Before (v0.3.x):**

```go
import (
    "github.com/digitallysavvy/go-ai/pkg/ai"
    "github.com/digitallysavvy/go-ai/pkg/telemetry"
    "go.opentelemetry.io/otel/trace"
)

tracer := trace.NewNoopTracerProvider().Tracer("my-app")

// v0.3.x: custom tracer passed per-call
result, err := ai.GenerateText(ctx, ai.GenerateTextOptions{
    Model:  model,
    Prompt: "Hello",
    ExperimentalTelemetry: &telemetry.Settings{
        IsEnabled: true,
        Tracer:    tracer, // custom tracer passed directly
    },
})
```

**After (v0.4.0):**

```go
import (
    "github.com/digitallysavvy/go-ai/pkg/ai"
    "github.com/digitallysavvy/go-ai/pkg/telemetry"
)

// v0.4.0: register integration once at startup
func main() {
    telemetry.RegisterTelemetryIntegration(telemetry.OTelTelemetryIntegration{})

    // All subsequent calls automatically use the registered integration
    result, err := ai.GenerateText(ctx, ai.GenerateTextOptions{
        Model:  model,
        Prompt: "Hello",
        // ExperimentalTelemetry still available for per-call control:
        ExperimentalTelemetry: &telemetry.Settings{
            IsEnabled:    true,
            RecordInputs: true,
            FunctionID:   "my-chat",
        },
    })
    if err != nil {
        log.Fatal(err)
    }

    fmt.Println(result.Text)
}
```

If you were not passing a custom `Tracer` and relied on the default global tracer, no changes are needed. The `OTelTelemetryIntegration` is registered globally by default and uses `otel.Tracer("ai-sdk")` when no custom tracer is provided.

The global registry supports multiple integrations via `AddTelemetryIntegration()` for fan-out to multiple backends. Use `ClearTelemetryIntegrations()` to reset, or pass `telemetry.NoopTelemetryIntegration{}` to `RegisterTelemetryIntegration()` to disable telemetry.

## New features

v0.4.0 also introduces several new capabilities:

- **Top-level Reasoning parameter** ŌĆö Configure reasoning effort (`low`, `medium`, `high`) directly on `GenerateText` and `StreamText` options instead of per-provider settings. See [Generating Text](/docs/ai-sdk-core/generating-text).
- **Deferred provider tool results** ŌĆö Providers can now mark tool calls as `ProviderExecuted: true` (e.g., Google code execution), returning results inline without requiring client-side execution. See [Tools and Tool Calling](/docs/ai-sdk-core/tools-and-tool-calling).
- **CustomContent and ReasoningFileContent types** ŌĆö New content block types for provider-specific content (`CustomContent`) and reasoning trace files (`ReasoningFileContent`). See [Generating Text](/docs/ai-sdk-core/generating-text).
- **Tool-level timeouts** ŌĆö Set a `Timeout` duration on individual tools to limit execution time independently of the overall request context. See [Tools and Tool Calling](/docs/ai-sdk-core/tools-and-tool-calling).
- **Embed/rerank callbacks** ŌĆö `OnStart` and `OnFinish` callbacks for `Embed`, `EmbedMany`, and `Rerank` operations, with provider options forwarding and HTTP response metadata. See [Embeddings](/docs/ai-sdk-core/embeddings).
- **New providers** ŌĆö [Prodia](/docs/providers/prodia) image generation provider and [KlingAI v3.0](/docs/providers/klingai) motion control for video generation.

## Migration checklist

### Code changes

- [ ] Update `StreamText` consumers that depend on mid-stream tool results to handle deferred execution
- [ ] Replace `xai.Provider.LanguageModel()` with `ChatCompletionsLanguageModel()` if you need Chat Completions API behavior
- [ ] Replace `grok-2` and `grok-2-vision-1212` model IDs with `grok-3` family models
- [ ] Add `Config: mcp.TransportConfig{Redirect: mcp.MCPRedirectFollow}` to MCP transport configs that need redirects
- [ ] Move custom `Tracer` setup to `telemetry.RegisterTelemetryIntegration()` at startup

### Testing

- [ ] Run `go build ./...` to verify compilation
- [ ] Run `go test ./...` to verify all tests pass
- [ ] Verify streaming consumers handle deferred tool execution correctly
- [ ] Test XAI integrations with new Responses API default
- [ ] Verify MCP transport behavior with redirect changes
